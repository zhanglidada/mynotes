### 1. Oracle（PGA 硬限制 + 主动杀死会话，可以借鉴先拒绝的思路）

**实例级上限**

- 参数：`PGA_AGGREGATE_LIMIT`（12c 引入），它是实例全部 PGA 的**硬性上限**。
    
- 实现逻辑：
    
    - Oracle 有一个后台进程 `CKPT` 以及内存管理器会持续统计所有服务进程和后台进程的 PGA 用量。
        
    - 当总 PGA 用量达到 `PGA_AGGREGATE_LIMIT` 时，不再允许任何进程扩大其“可自动调整的工作区”（如排序区、哈希区），后续需要此类内存的操作会直接报错 `ORA-04036`。
        
    - 如果总用量仍继续上升（例如某些会话持有大量“不可自动调整”内存未释放），Oracle 会直接**向占用 PGA 最多的会话发送 SIGKILL**，并立即释放其持有的全部内存，直到总量回到限制值以下。Alert log 中会打印 `PGA memory used by the instance exceeds PGA_AGGREGATE_LIMIT`。
        
- 这是典型的“~={orange}先拒绝再杀进程=~”模式，本质是**用强制终止少数肇事会话来保全整个实例**。
    

**会话级限制**

- Oracle 没有直接的单会话总内存硬上限参数。但 `PGA_AGGREGATE_LIMIT` 是一个强制内存清理机制，内存占用最大的会话会被优先终止，起到了“大会话自动清理”的效果。

**预警/主动干预**

- 一旦触及 `PGA_AGGREGATE_LIMIT`，实例立刻进入保护状态（报错 + 杀会话），相当于以“错误”作为预警信号，并自动执行干预。

### 2. SQL Server（max server memory + 资源调控器，可以借鉴）

**实例级上限**

- 参数：`max server memory`，严格限定 SQL Server 进程可提交的**总内存**（含缓冲池、查询执行内存、计划缓存等一切）。
    
- 实现逻辑：
    
    - SQL Server 内部有内存管理器，会实时跟踪所有内存分配者的总量。当进程内存逼近 `max server memory` 时，不会立刻拒绝分配，而是：
        
        - 强制检查点，将脏页写出，释放缓冲池页面。
            
        - ~={orange}减少查询执行内存授予，使得需要大量内存的查询等待或超时。=~
            
        - ~={orange}必要时缩小内部缓存（如计划缓存）。=~
            
    - 若所有释放手段用尽后仍需要内存，新的内存请求会失败，查询返回内存不足错误（例如 `8645` 超时）。
        
- 它不像 Oracle 那样主动杀会话，而是通过~={orange}**收缩已有缓存 + 限制新授予**来维持水位，保护 OS 不被撑爆=~。
    

**会话/工作负载级限制**

- 通过 **Resource Governor（资源调控器）** 将工作负载分配到“资源池”，并对池设置：
    
    - `MAX_MEMORY_PERCENT`：该池可占用的总查询执行内存上限（实例级别可用内存的百分比）。
        
    - `REQUEST_MAX_MEMORY_GRANT_PERCENT`：单个请求最大可授予的内存百分比。
        
- 实现方式：
    
    - 资源池由 SQL OS 调度层强制执行。当某请求需要内存执行排序/哈希时，会先向资源池申请内存授予。如果该池已使用量超过其上限，或单个请求超出其允许授予上限，则查询会进入等待队列，直到有内存释放或超时。
        
    - 这样就实现了**会话/操作级硬限制**，一条烂 SQL 不会耗尽所有查询执行内存。
        

**阈值预警**

- 当物理内存压力过大（默认达 80%）时，SQL Server 会主动缩小缓冲池和内部缓存，可通过性能计数器 `Memory Manager: Memory Grants Pending` 等监控。
    
- 有内置的 `sp_configure 'show advanced options'` 及 `Resource Governor` 相关的 DMV 可配置告警。

### 3. TiDB（硬上限 + 按百分比主动杀查询，直接强杀不建议参考）

**实例级上限**

- 参数：`tidb_server_memory_limit`（或之前称为 `server-memory-quota`），设置 TiDB 服务器进程的**总内存软硬兼施的阈值**。
    
- 实现逻辑：
    
    - TiDB 用 Go 编写，运行时内置了内存跟踪，能够统计所有查询、缓存、内部数据结构使用的堆内存。
        
    - 当总体内存使用量接近该限制时（比如达到 **70%**），TiDB 开始主动采取行动：
        
        - 记录报警日志，提示内存压力。
            
        - 从所有正在执行的 SQL 中，找出**内存使用量最大的 SQL**，将其强制取消（KILL），释放内存。
            
    - 若内存持续升高，会持续杀掉消耗最多的 SQL，直到总内存回落。
        
- 与 Oracle 的做法高度相似，但 TiDB 的阈值是百分比（可配置的触发水位），而不是单一线上的硬碰硬。

### 4. Informix（会话硬限制，缺乏实例上限，没啥借鉴意义）

**实例级上限**

- Informix 没有严格意义上的实例内存硬上限参数。通常依赖操作系统 `ulimit` 或 cgroup 来限制整个 `oninit` 进程的内存。
    

**会话级限制**

- 参数：`SESSION_LIMIT_MEMORY`（可在 `onconfig` 或会话级设置）。
    
- 实现逻辑：
    
    - 每个会话在分配内存时，数据库服务器会检查该会话已使用的私有内存是否超过此值。
        
    - 一旦超过，该会话的当前事务会被回滚，并报错 `-1723` 或 `-1724`，会话被断开，内存释放。管理员用户（DBA）通常不受此限制。
        
    - 它是一个**硬限制**，直接终止肇事会话，简单粗暴但有效。
        

**预警**

- 当会话内存超限时，数据库会记录事件告警，并可触发 `ALARMPROGRAM` 脚本。

### 5. DB2（实例内存硬顶 + 应用级堆限制，可以借鉴思想）

**实例级上限**

- 参数：`INSTANCE_MEMORY`（或自动模式），设定整个实例能使用的**最大内存**，包括数据库共享内存和各种代理私有内存。

	参数限制范围在机器总内存的75%到95%之间

- 实现逻辑：
    
    - DB2 内存管理器会持续统计所有内存使用。当总用量达到 `INSTANCE_MEMORY` 时，**新的内存分配请求会被直接拒绝**，操作返回错误（如 `SQL0289N`），但**不会强行终止已有会话**。
        
    - 这确保了实例内存永远不会越界，是一种“拒绝服务而非杀死进程”的策略。
        

**会话/应用程序级限制**

- 数据库级别参数 `APPL_MEMORY` 限制单个连接能使用的**代理私有内存总量**（如排序堆、语句堆等）。
    
- 实现方式：代理进程在执行 SQL 时，其私有内存申请需在 `APPL_MEMORY` 内，超限则请求失败，当前语句报错，不影响其他连接。

### 6. MongoDB（缓存限制，无会话级）

**实例级上限**

- 参数：`storage.wiredTiger.engineConfig.cacheSizeGB`，指定 WiredTiger 引擎内部缓存的最大大小，通常设为物理内存的 60%~80%。
    
- 实现逻辑：
    
    - WiredTiger 使用自己管理的内存，当缓存使用接近上限时，会触发**主动淘汰（Eviction）**，将冷数据页写出磁盘，腾出空间。
        
    - 该参数只限制**缓存部分**，不包括连接线程栈、网络缓冲区、JavaScript 引擎等内存。因此 MongoDB 的整体进程内存可能超出 `cacheSizeGB`。
        
- 实际生产环境通常结合 cgroup/容器对 `mongod` 进程进行总内存硬限制。
    

**会话/操作级限制**

- 无直接限制单会话或单查询的总内存。仅通过以下操作级参数控制：
    
    - `internalQueryExecMaxBlockingSortBytes`：排序内存上限，超限报错。
        
    - `maxIndexBuildMemoryUsageMegabytes`：索引构建内存上限。
        
    - 聚合管道可以通过 `allowDiskUse: true` 将大部分操作溢写到磁盘。
        
- 不能全局防止一个连接执行多条大操作而堆积内存。
    

**预警**

- 当缓存使用超过 **80%** 时会触发 Eviction 加重，如果达到 **95%** 且出现 SWAP 使用超过 5%，则视为严重瓶颈，可通过 `serverStatus` 监控。

### 7. MySQL 8.0（连接级硬限制，依赖 OS 实现实例限制）

**实例级上限**

- MySQL 内核**没有**参数直接限制 `mysqld` 进程总内存。通常做法是：
    
    - 在云 RDS 中，平台通过 cgroup 强行限制实例内存。
        
    - 在自建环境中，配置 `innodb_buffer_pool_size`、`max_connections` 等间接控制，或通过 Docker/Systemd 的 `MemoryMax` 设定 cgroup 上限。
        
- 因此，纯粹原生 MySQL 无法自我保护，超限时会被 OS OOM Killer 杀死，存在主从切换风险。
    

**会话/连接级限制**

- 从 8.0.28 开始引入两个全局变量：
    
    - `connection_memory_limit`：**单个用户连接**允许使用的最大内存（包括临时表、内部缓冲区等）。
        
    - `global_connection_memory_limit`：**所有用户连接**合计的内存硬上限。
        
- 实现逻辑：
    
    - 服务器会跟踪每个连接分配的内存总量。当某个连接申请内存将导致其自身或全局总量超出限制时，内存分配函数返回错误，SQL 查询失败，并报错 `ER_DA_CONN_LIMIT: Memory limit exhausted`。
        
    - 连接本身不会被杀死，只是查询报错。
        
- 这是 **DB2 风格**的保护：拒绝分配，保留连接，由应用重试。
    

**预警**

- 无内置使用率预警。完全依赖外部监控（如 Prometheus + `memory_global_by_current_bytes` 等表）。

### 8. PostgreSQL（操作级软限制，完全依赖 OS 总控）

**实例级上限**

- 原生无参数限制 PostgreSQL 进程总内存。通常通过：
    
    - 操作系统 `ulimit -v` 限制单个进程虚拟内存（但 PG 是多进程，每个后端进程都有自己的限制，难以控制总和）。
        
    - 容器/cgroup 的 `memory.limit_in_bytes` 限制整个进程组的总内存。
        
- 超限时由 OOM Killer 处理，可能导致随机进程被杀，是 PG 在内存保护上的最大痛点。
    

**操作级限制**

- `work_mem`：指定单次排序或哈希操作可使用的内存上限。超出后，操作改用磁盘临时文件。
    
- `maintenance_work_mem`：类似，用于 VACUUM 等维护操作。
    
- `hash_mem_multiplier`：控制哈希操作可额外使用的倍数。
    
- 实现逻辑：  
    执行器在分配排序/哈希内存时，会检查当前使用量，一旦达到 `work_mem`，就不再申请新内存，转为将数据分批写出到磁盘。**这是操作级软限制，不报错、不杀查询**。
    
- 问题：一个会话可以同时运行多个排序/哈希操作，每个都可用到 `work_mem`，所以实际会话内存可能是 `work_mem` 的很多倍。
    

**预警**

- 无内置预警，只能监控 `pg_stat_activity` 和相关系统视图构建外部告警。

### 总结：实现逻辑分类

|保护策略|代表数据库|内核实现方式|
|---|---|---|
|**硬上限拒绝分配**|DB2 (`INSTANCE_MEMORY`)、MySQL 8.0 (`connection_memory_limit`)|在内存分配入口检查计数器，超限返回错误|
|**硬上限 + 主动杀会话**|Oracle (`PGA_AGGREGATE_LIMIT`)、TiDB (`tidb_server_memory_limit`)、Informix (`SESSION_LIMIT_MEMORY`)|后台监控总用量，超阈值后选择内存占用最大的任务/会话发送取消或终止信号|
|**上限软控 + 查询降级**|SQL Server (`max server memory` + 资源调控器)、MongoDB (缓存淘汰)|内存管理器通过缩小缓存、减少授予、启用磁盘溢写来维持水位，极端情况才报错|
|**操作级软限制，无全局保护**|PostgreSQL (`work_mem`)、MongoDB (排序限制)|执行器仅在单个操作超过内存配额时切换为磁盘存储，不阻止多个操作累加导致内存耗尽|
